# RFC: ai-barracks v1.1.0 — Skills as First-Class Capability

> **Status**: Draft (2026-05-08)
> **Origin**: Promoted from `ai_barracks_management` 배럭 wiki (session claude-20260508-1218)
> **Targets**: ai-barracks framework 1.1.0, ai-barracks-cc 1.1.0
> **Depends on**: [Anthropic Agent Skills (2025-10-16)](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills), [GitAgent Protocol (GAP 0.1.0)](https://www.gitagent.sh/)

## Goal

ai-barracks 배럭이 **Anthropic Agent Skills 표준(2025-10) 및 GitAgent GAP 0.1.0과 호환되는 invocable skill 카탈로그**를 보유하도록 한다. wiki(passive 지식)와 직교하는 능동적 capability 레이어를 도입.

## Non-Goals (v1.1 범위 밖)

- subagent dispatch (`agents/` 디렉터리) — v1.2 예정
- 배럭 간 위임(`delegation`) — v2.0 예정
- compliance/risk_level/HITL 명시화 — v2.0 예정
- 기존 wiki/sessions/GROWTH 구조 변경 — 유지

## Spec

### S1. Directory Layout

각 배럭에 `skills/` 디렉터리를 둔다:

```
{barrack}/
├── agent.yaml
├── SOUL.md
├── RULES.md
├── GROWTH.md
├── wiki/                  # 기존
├── sessions/              # 기존
└── skills/                # NEW
    └── <skill-slug>/
        ├── SKILL.md       # 필수
        ├── scripts/       # 선택 (실행 스크립트)
        └── references/    # 선택 (progressive disclosure 추가 파일)
```

### S2. SKILL.md Frontmatter

Anthropic 표준 + ai-barracks 확장:

```yaml
---
# 필수 (Anthropic 표준)
name: council
description: "<discovery-trigger phrase — when to invoke>"

# 선택 (Claude Code slash command 호환)
argument-hint: "[topic] [-m mode] [-r rounds]"
allowed-tools: Bash(./scripts/council.sh *), Read, Write

# 선택 (ai-barracks 확장 — 무시되어도 동작)
aib_version: "1.1"
upstream: "ai-barracks/scripts/council.sh"  # 외부 스크립트 참조 시
growth_origin: "manual"  # manual | growth-auto-generated
---
```

규칙:
- `name`은 디렉터리 슬러그와 일치해야 함 (`skills/council/` ↔ `name: council`)
- `description`은 invoke 결정의 1차 메커니즘. 동작/트리거를 명확히 명시
- `aib_version`은 호환성 마커 — 1.1 이전 클라이언트는 무시

### S3. agent.yaml 확장

기존:
```yaml
skills:
  - council
```

신규(둘 다 허용):
```yaml
skills:
  discovery: auto      # skills/ 자동 스캔. 또는 explicit
  enabled:             # discovery: explicit일 때만 의미 있음
    - council
  external:            # 다른 위치(upstream framework 등) 스킬 참조
    - upstream:ai-barracks/scripts/council
```

backward compat: 기존 `skills: [council]` 리스트 형식은 `discovery: explicit, enabled: [council]`로 해석.

### S4. GROWTH 트리거 — Skill Candidate Detection

GROWTH.md의 Decision Table에 신규 행 추가 제안:

| 세션 중 이벤트 | 기록 위치 | 예시 |
|---------------|-----------|------|
| **동일 워크플로 3회+ 반복 (지시→도구 사용 시퀀스 동일)** | `sessions/{id}.md` § Skill Suggestions | "PR 생성 5회 반복 → `/skill:open-pr` 후보" |

세션 종료 시 audit에서 skill 후보가 누적되면 사용자가 검토 후 `skills/<slug>/SKILL.md`로 승격. **에이전트가 skills/를 직접 자동 생성하지 않는다** (SOUL.md 자기수정 금지 원칙과 동일 보호).

### S5. 호출 규약

- Claude Code: `Skill` tool로 invoke (이미 표준)
- 슬랙 봇/외부 클라이언트: `/skill:<name> [args]` 슬래시 명령 (선택적 호환 레이어)

## S6. Loading (added v1.2.0)

Catalog (S1–S3) is necessary but not sufficient — the registered skills must reach the running session. v1.2.0 introduces the loading layer.

### S6.1 Wirings

| ID | CLI | Mechanism | Storage |
|----|-----|-----------|---------|
| W1 | Claude Code | Project-local skill discovery via `.claude/skills/<n>` (relative symlink to `../../skills/<n>`) | runtime artifact, gitignored |
| W2 | Gemini, Codex (no native invoke) | Auto-managed `<!-- AIB:SKILLS:START --> ... <!-- AIB:SKILLS:END -->` block in `CLAUDE.md` = `GEMINI.md` = `AGENTS.md` | committed |

W2 also ships in CLAUDE.md so Claude has redundant text-level discovery in addition to W1's native registration. The three .md files remain byte-identical (single source of truth).

### S6.2 Master / Drift Model

- `aib sync` is the **static master**. It (re)materializes W1 + W2 from the current `skills/` catalog. Idempotent: identical inputs ⇒ byte-identical outputs.
- `aib start <client>` runs `check_skills_drift` and surfaces a non-blocking warning if the catalog and materialized wirings have diverged. It does **not** auto-regenerate (preserves the user-explicit-action principle).
- `aib skills doctor` includes drift in its checks; `aib skills check` exposes drift as a standalone diagnostic.

### S6.3 Filesystem Contract

- W1 symlinks are always relative (`../../skills/<n>`) so the link survives moves of the barrack root.
- Orphan symlinks (target skill removed) are cleaned by `sync_claude_skills_symlinks` before new links are created.
- W2 user content above and below the marker block is preserved across regenerations (same extract-and-rebuild pattern as `inject_protocol`).

### S6.4 Compatibility

- v1.1.x clients reading a v1.2-synced barrack will see the W2 SKILLS marker block as plain markdown — harmless, no behavior change.
- v1.2 clients reading a v1.1-synced barrack (no W2 block, no W1 symlinks) will materialize both on first `aib sync`. No migration command needed.

## Open Questions

- Q1. Skill 자동 디스커버리 시 token 비용? Anthropic은 metadata만 시스템 프롬프트 로드(progressive disclosure). ai-barracks도 description만 wiki Index처럼 노출하는 게 맞음 — 구현 시 확인 필요
- Q2. external skill 참조(`upstream:`)의 구체 해석 규칙? 심볼릭 링크 vs 동적 로드 vs 빌드타임 복사 → **결론**: 별도 메커니즘 도입하지 않는다. `templates/skills/`가 `aib sync`로 모든 배럭에 복사되는 것이 곧 표준 전파 경로. `upstream:` 필드는 *문서적 메타데이터*(어디서 유래했는지)일 뿐 런타임 의미 없음
- Q3. SOUL.Expertise와 skills 카탈로그의 관계? Expertise = 선언된 정체성, skills = 검증된 capability. 동기화 메커니즘은 v1.2에서 검토
- Q4. council처럼 multi-CLI 호출하는 스킬은 보안/권한 정책을 어떻게 명시? `allowed-tools` 만으로 충분한가? — Anthropic 표준이 `allowed-tools` Bash 패턴 매칭이라 호환성은 OK. 추가 격리(network/fs)는 v2.0 compliance로 이연
- Q5. **Cross-client 동작?** Claude Code는 `Skill` tool로 native invoke. Gemini CLI/Codex CLI는 native Skill 미지원 — SKILL.md는 *문서로* 읽히고, 구체 호출은 SKILL.md 본문의 `argument-hint` + 명시된 bash 명령에 따라 수동/슬래시 실행. v1.1은 "skills/는 client-agnostic 지식 저장소"라는 위치만 확정하고, 클라이언트별 어댑터는 client별 책임으로 둔다
- Q6. **빈 `skills/` 배럭 허용?** 허용. `discovery: auto`라도 디렉터리가 없거나 비면 단순히 카탈로그 0개로 처리. 신규 배럭은 `templates/skills/council/`만 시드
- Q7. **Skill 슬러그 충돌?** 한 배럭 내 동일 슬러그 금지(파일시스템이 강제). 다른 배럭 간은 무관 — 배럭 격리 원칙 유지
- Q8. **버전 호환?** 1.0.x 클라이언트는 새 `skills:` 객체 스키마를 무시(YAML 키 unknown 처리). 1.1 클라이언트는 리스트·객체 둘 다 파싱. agent.yaml `aib_version` 필드로 감지
